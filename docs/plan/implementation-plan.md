# 地価×駅距離×乗降客数 分析・予測マップ 実装計画

## 作成日時

2026-07-04

## やりたいこと

国土数値情報の地価公示（L01）・駅データ（N02）・駅別乗降客数（S12）を組み合わせ、「最寄駅までの距離」「駅の乗降客数」等の特徴量から地価を回帰予測するWebマップを構築する。

- 実測地価と予測地価の**乖離率**（割安・割高）を地図上で色分け表示する
- 地図上の任意地点をクリックすると、その場で予測地価を返す（推論API）
- 既存サービス（全国地価マップ等）は「地価の表示」に留まっており、**駅距離・乗降客数との相関分析・予測モデルまで踏み込んだものは存在しない**ため、分析レイヤーで差別化する
- 学習目的：geopandasによる空間データ処理、LightGBMによる回帰モデル、TerraformによるIaC（汎用的なIaCスキルの習得）、サーバーレスML推論

---

## 基本方針

**Phase 1（案A：完全静的構成）→ Phase 2（案C：Terraform + AWS構成、案Bの推論APIを機能として吸収）** の2段階で進める。

| 方針                                                  | 理由                                                                                                                              |
| ----------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------- |
| 前処理・MLコードはPhase間で完全共通化                 | Phase 1の成果物（前処理スクリプト・学習済みモデル）をPhase 2でそのまま流用し、二重開発を避ける                                    |
| Phase 1で最短公開                                     | 「動く分析マップ」を早期に得てモチベーションを確保。地価公示は年1回更新のため、静的構成でも実用上のミスマッチがない               |
| Phase 2でAWS学習に集中                                | アプリの機能はPhase 1で完成済みのため、Phase 2ではTerraform・サーバーレス・パイプライン自動化という「インフラの学び」に専念できる |
| 案Bの推論APIはPhase 2のLambda + API Gatewayとして実装 | 独立したFly.io構成を挟まず、AWS上で「任意地点クリック→予測」機能を追加する                                                        |

## 具体的な実装手順

### Phase 1: 静的構成（案A）

1. **データ取得**
   - 地価公示 L01、駅データ N02、駅別乗降客数 S12 を国土数値情報からダウンロード（GeoJSON または Shapefile）
2. **前処理パイプライン構築**（Python + geopandas）
   - S12の駅ライン（線ジオメトリ）を重心化してポイント化し、駅コードで乗降客数を結合
     - 名寄せは**駅グループ単位**（確定）：S12の重複コードをキーにgroupbyし、乗降客数はグループ合算する。駅名は表示専用とし、識別には重複コードを使う
   - S12は年度ごとに4カラム（重複コード・データ有無コード・備考・乗降客数）が横に増える構造のため、対象年度のカラム抽出処理を実装
     - 対象年度は**2024年度**（確定。理由は「要確認事項」の確認結果を参照）。年度は設定値として外出しし、次年度更新時にコード変更なしで切り替えられるようにする
   - `sjoin_nearest` で各地価ポイントに最寄駅・駅距離（直線距離）を付与
     - 駅距離はL01付属の「最寄駅までの道路距離」ではなくN02由来の直線距離を自前計算する。道路距離は推論API（任意地点）で再現できず、学習時と推論時で特徴量の定義を一致させる必要があるため
     - L01付属の道路距離は前処理の妥当性検証に使う（自前計算の直線距離 ≦ L01の道路距離 となるはず。大きく矛盾する地点は名寄せ・結合のバグを疑う）
   - 特徴量テーブル（最寄駅距離、乗降客数、用途地域、容積率、市区町村等）をParquet/CSVで出力
3. **モデル学習**（scikit-learn / LightGBM）
   - 地価（円/m²、対数変換推奨）を目的変数とした回帰
   - ホールドアウトまたは交差検証でRMSE / R²を確認
   - 予測値・乖離率を付与したGeoJSONを出力（全国約2.6万地点、数MB想定）
4. **フロントエンド構築**
   - React + TypeScript + Vite + MapLibre GL JS
   - 背景地図は国土地理院タイル
   - 乖離率のdata-driven styling（割安=青、割高=赤）、地点クリックでポップアップ（実測・予測・特徴量表示）
   - 出典・免責表記を実装：地図クレジット（attributionControl）に国土数値情報（加工した旨つき）と地理院タイルの出典を併記し、「このサイトについて」ページに独自推計である旨・学習目的である旨を記載（文言は「要確認事項」の確認結果参照）
   - 表示が重い場合はtippecanoeでPMTiles化
5. **公開**
   - 構築前にAWSアカウントの初期設定を行う：リージョンは東京（ap-northeast-1）、AWS Budgets（月1,000円・実績80%・予測100%でメール通知）とCost Anomaly Detectionを設定（詳細は「要確認事項」参照）
   - S3 + CloudFrontへデプロイ（Phase 1ではAWSコンソール / AWS CLIによる手動構築で最短公開を優先）
   - OAC（Origin Access Control）でS3バケットを非公開のまま配信する
   - この構成はPhase 2で `terraform import` によりコード管理下へ移す前提とする

### Phase 2: Terraform + AWS構成（案C、推論API＝旧案Bを吸収）

1. **Terraformプロジェクト初期化**
   - リモートstate管理：state用S3バケットを最初に構築し、ロックは**S3ネイティブロック（`use_lockfile = true`）**を使う（DynamoDBテーブルは作らない。確定理由は「要確認事項」参照）
   - Terraformは最新安定版をmiseで固定し、`required_version` と `.terraform.lock.hcl` をGit管理する
   - モジュール分割：`frontend` / `inference` / `pipeline` の3モジュール構成（CDKのスタック分割に相当）
2. **フロント配信のTerraform化**（frontendモジュール）
   - Phase 1で手動構築したS3 + CloudFrontを `terraform import` でコード管理下に移す（importの実践も学習対象とする）
   - import が煩雑な場合はTerraformで新規構築し、DNS切り替えで移行する
3. **推論APIの実装**（inferenceモジュール ＝ 旧案B）
   - Lambda（コンテナイメージ、LightGBM + 学習済みモデル同梱）+ API Gateway
   - `POST /predict {lat, lng}` → サーバー側で最寄駅検索＋特徴量生成＋推論
   - 最寄駅検索は駅ポイントを同梱しscipy KDTree等で実装（データ量的にDB不要）
   - フロントに地図クリック→API呼び出し→予測表示の機能を追加
4. **データパイプラインの自動化**（pipelineモジュール）
   - S3にrawデータ配置 → 前処理・再学習（Lambda、重い場合はFargate）→ 成果物をS3へ
   - EventBridgeスケジュール or Step Functionsでオーケストレーション（地価公示の年1回更新に対応）
   - 年次更新ではL01の年度とS12の対象年度を**ペアで更新**する（L01は3月、S12は4月頃に新年度版が公開されるため、更新タイミングは両者が揃う4月以降とする）
5. **運用整備**
   - CloudWatchによるログ・メトリクス確認
   - 予算アラートはPhase 1の公開前に設定済みのため、Phase 2ではリソース追加（Lambda・ECR等）に伴う上限額の見直しのみ行う

## テスト方針（TDDの適用範囲）

TDDは「入力と期待出力を事前に定義できる」レイヤーに絞って適用する。モデル精度のように実験しないと正解が分からない部分は、TDDではなく実験管理・回帰テストで担保する。

| レイヤー            | 方針                                                                                                              |
| ------------------- | ----------------------------------------------------------------------------------------------------------------- |
| 前処理（geopandas） | **TDDを本格適用**。「リスクと注意点」の各項目をテストケース化する                                                 |
| 学習コードの配管    | 仕様固定のための単体テスト（対数変換の往復、出力GeoJSONのスキーマ、データ分割の漏洩チェック等）                   |
| モデル精度          | テスト対象外。メトリクス記録（RMSE / R²）で実験管理。モデル更新時は前バージョンとの劣化チェック（回帰テスト）のみ |
| 推論API（Phase 2）  | **TDDを適用**。LambdaハンドラをAWS非依存の関数に分離し、ローカルでテスト可能にする                                |
| フロント            | ロジック関数（乖離率→色変換、ポップアップ整形等）のみ単体テスト。地図描画は目視確認                               |
| Terraform           | `terraform plan` の目視確認を基本とし、テストは導入しない                                                         |

### テストシナリオ（Gherkin記法・設計ドキュメント）

前処理・推論APIの具体的なテストシナリオはGherkin記法で以下に整理済み。**実行はせず**（pytest-bdd等は導入しない）、pytestのテストケース作成時のテストリストとして使用する。シナリオとテスト関数を1対1で対応させ、docstringにシナリオ名を記載する。

- 前処理：`docs/private/plan/test/preprocessing.feature`
- 推論API：`docs/private/plan/test/inference-api.feature`

### 実装上の注意

- テストフィクスチャは数行の小さなGeoJSON / DataFrameを用意し、実データに依存しない
- 推論・フロントとも「テスト可能なロジック」と「外界との接続（AWS SDK・地図描画）」を分離する設計を最初から意識する

## リスクと注意点

- **S12のデータ構造の癖**：年度ごとにカラムが横に増える形式。データ有無コード（非公開駅あり）の考慮を怠ると乗降客数の欠損・ゼロを誤学習する
- **駅ジオメトリがライン**：重心化の際、地下鉄等で複数事業者の同名駅が別レコードになるケースがある。名寄せは駅グループ単位（重複コード）で行うと決定済み（「要確認事項」参照）。別名の実質乗換駅（新宿と西武新宿等）が統合されない点は割り切る
- **任意地点予測の特徴量問題**（Phase 2）：クリック地点には用途地域・容積率が存在しない。学習時から「オンラインで取得可能な特徴量のみのモデル」を別途用意するか、用途地域データ（A29等）を推論時に空間検索する設計が必要。**Phase 1の時点でモデルを2系統に分けておくと手戻りがない**
- **Lambdaイメージサイズ**：geopandas + LightGBMを同梱すると1GB超になりがち。推論時はgeopandas不使用（KDTree + numpyのみ)に絞るなど依存の分離が必要
- **CloudFrontの転送量課金**：Cloudflare Pagesと異なり帯域は従量課金。全国GeoJSONが数十MB規模になるとアクセス増で課金が目に見えるため、PMTiles化（必要な範囲のみRange Requestで取得）によるサイズ削減の意義が大きい
- **Terraformのstate管理**：stateファイルにはリソース情報（場合によりシークレット）が平文で入るため、リモートstate用S3バケットは暗号化・バージョニング有効化し、**stateファイルは絶対にGitにコミットしない**（`.gitignore`に`*.tfstate*`を追加）。state用バケット自体はTerraform管理外で先に手動作成する「鶏と卵」問題に注意
- **CDKとの記述量差**：TerraformはCDKのL2コンストラクトのような高レベル抽象がなく、IAMロール・ポリシー等を明示的に書く必要がある。学習効果は高いが、Lambda + API Gateway周りの記述量は増える前提で見積もる
- **予測の見せ方**：「割安・割高」表示は不動産の投資判断と誤解されうる。学習プロジェクトである旨を明記し、予測地価・乖離率は**本サイト独自の推計であり国の公表値ではない**ことを明示する（利用規約の「国が作成したかのような態様での公表」禁止にも対応。出典表記の確定文言は「要確認事項」参照）
- **精度の期待値管理**：地価は駅距離・乗降客数だけでは説明しきれない（商業集積、地形、規制等）。R²が伸びなくても「乖離の可視化」自体に価値がある、という位置づけを維持する

  - **確認済み（2026-07-07）**：最新は令和8年（2026年）版（2026年3月更新）。用途地域・容積率（上限）・建蔽率（上限）のカラムはいずれも存在し、GeoJSON / Shapefile形式で取得可能。設計の前提は成立
  - 属性は `L01_xxx` 形式の連番コードで管理され年度により構成が変わりうるため、実装時にダウンロードした年度の属性コード表で列番号を確定させること
  - L01には「最寄駅名」「最寄駅までの道路距離」属性も存在する（下記「前処理パイプライン構築」の検証に使用）
  - **実装時確認済み（2026-07-07、L01-26のGML要素名と突き合わせ）**：地価=`L01_008`、用途地域=`L01_051`、容積率=`L01_058`、市区町村コード=`L01_001`、最寄駅名=`L01_048`、道路距離=`L01_050`。文字列属性の欠損表現は`_`。マッピングは `pipeline/src/landprice/config.py` の `L01ColumnMap` に設定済み
  - 注意：L01の「最寄駅」は鉄道がない地域ではバス停等が入る（例：石垣島のバスターミナル）。道路距離との整合検証では該当地点が要確認としてレポートされるが想定内
- [x] S12の対象年度をどれにするか（最新2024年度版か、地価公示の基準日と合わせるか）
  - **確認済み（2026-07-07）**：最新は2024年度（令和6年度）版（2026年4月公開）。乗降客数は2011〜2024年度の14年分を収録
  - **対象年度は2024年度を採用**。地価公示令和8年（基準日2026年1月1日）に対し入手可能な最新であり、2025年度分は統計のタイムラグで未公開のため、「最新を使う」「基準日に近づける」いずれの方針でも同じ帰結になる。コロナ禍（2020〜2022年度）の乗降客数の歪みも回避できる
  - 次年度以降の再学習では**L01の年度更新に合わせてS12の対象年度もペアで更新**する（2025年度版は2027年春公開見込み）
  - **実装時確認済み（2026-07-07）**：2024年度の乗降客数を収録しているのは**S12-25**（S12_058〜061）。S12-24は2023年度までしか収録していないため使用しない。パイプラインはS12-25を入力とする
- [x] 駅の名寄せ単位（事業者別 or 駅グループ単位）
  - **確定（2026-07-07）：駅グループ単位で名寄せする**。S12の重複コード（「同名かつ300m以内」の駅を1グループとし、グループ重心に最も近い駅コードを持たせた既存属性）をキーに、乗降客数をグループ合算する
  - 理由：事業者別のままでは駅の集客力が事業者ごとに分割されて過小評価され、どの事業者のポイントが最寄りに選ばれるかで乗降客数が大きくブレて雑音になるため。グループ合算なら「駅全体の集客力」として特徴量の意味が安定する
  - 駅名は表示専用（ポップアップ等）とし、識別キーには重複コードを使う（同名でも300m超離れた別グループの駅が存在しうるため、駅名はユニークキーにならない）
  - 割り切り：「新宿」と「西武新宿」のような**別名**の実質乗換駅は統合されない（同名+近接が基準のため）
  - 重複コードの具体的なコード値の意味（代表・非代表の区別等）は、実装時に対象年度の製品仕様書・コードリストで確定させる
  - **実装時確認済み（2026-07-07、S12-25のXSDで確認）**：本計画で「重複コード」と呼んでいた名寄せキーは、実データでは独立属性の**グループコード（`S12_001g`）**。年度別に存在する「重複コード」は計上位置フラグ（1=当該路線駅に記載、2=他路線駅に記載、3=駅なし）で、**1のレコードのみ乗降客数の有効値**として扱い二重計上を防ぐ。データ有無コードは1=データ有、2=データなし、3=非公開、4=駅なし
- [x] Phase 2で使うAWSリージョン・アカウントの予算アラート設定
  - **確定（2026-07-07）：リージョンは東京（ap-northeast-1）**。推論APIのレイテンシ最小化のため。例外として、将来CloudFrontにカスタムドメインを付ける場合のACM証明書のみus-east-1で発行する
  - **予算アラート：AWS Budgetsで月予算1,000円**を設定し、**実績値80%（800円到達）・予測値100%（月末着地見込みが1,000円超）**の2段階でメール通知。あわせてCost Anomaly Detection（コスト異常検知）を有効化する。Budgetsは2つまで無料のため、いずれも無料枠内
  - Budgetsは通知であって自動停止ではないため、アラート受信時は手動でリソースを確認・停止する運用とする
  - **設定タイミングはPhase 1の公開前**。公開方式をS3 + CloudFrontとしているため、AWS利用前に設定する
- [x] Terraformのバージョンとロック方式（DynamoDBロック or Terraform 1.10以降のS3ネイティブロック）
  - **確定（2026-07-07）：S3ネイティブロック（`use_lockfile = true`）を採用**。DynamoDBテーブルは作成しない。従来の `dynamodb_table` は公式に非推奨化されており（Terraform 1.12以降で警告、将来削除予定）、今から構築するなら新方式が標準のため。state管理の構成が「S3バケット1つ」で完結する
  - **Terraform本体は着手時点の最新安定版を採用**（S3ネイティブロックがGAになった1.11以上を満たすこと）。`required_version` でプロジェクトに明示し、プロバイダのバージョンは `.terraform.lock.hcl` をGit管理して固定する
  - **ツールのバージョン管理はmiseに一本化**：miseでNode.js・Terraform・uvのバージョンを固定する。Python本体と依存ライブラリの解決はuvに任せ、miseとuvの役割重複を避ける
  - 注意：Web上の教材はDynamoDBロック前提のものが大半のため、`use_lockfile` に読み替えて学習する
- [x] 国土数値情報の利用規約（出典明記の文言・商用でない旨の確認）
  - **確認済み（2026-07-07）**：国土数値情報ダウンロードサイトは**公共データ利用規約（PDL1.0）準拠**（2026年3月23日施行）。出典を記載すれば複製・加工・公開は自由。L01は2019年以降**CC BY 4.0のオープンデータ**であり、**「商用でない旨の表明」は規約上の要件ではなくなった**（旧規約の非商用限定を前提とした本項目の後半は解消）
